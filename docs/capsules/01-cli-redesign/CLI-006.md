## 1. Executive Context

**Product Intent:** Mutations against Shopify (create, update, delete) have real consequences — they affect live stores. The CLI must protect users from accidental mutations with confirmation prompts and provide AI agents with a safe `--dry-run` mode to validate before executing.

**System Position:** Hooks into `toolToCommand()` handler in `src/cli/converter/tool-to-command.ts`. Checks tool scopes for `write_*` to determine if mutation safety applies.

## 2. Feature Specification

**What Exists Today:**
- Tool scopes array on each ToolDefinition (e.g., `["write_products"]`)
- `--dry-run` and `--yes` flags defined in global flags (CLI-003) but with no handler logic
- No confirmation prompts anywhere in the CLI

**What Must Be Built:**
1. Mutation detection: check if tool has any `write_*` scope → it's a mutation
2. `--dry-run` handler: validate input via Zod, show validated input + intent as JSON, DO NOT call ToolEngine.execute()
3. Interactive confirmation: if tool is a mutation AND stdout is TTY AND `--yes` not set, prompt `⚠ This will <action> <domain>. Continue? [y/N]`
4. `--yes` flag: skip confirmation (for CI/agents, also auto-set when not TTY)
5. Dry-run output format:
   ```json
   {
     "action": "create_product",
     "domain": "products",
     "input": {"title": "New Widget"},
     "confirmed": false,
     "message": "Would create product. Run without --dry-run to execute."
   }
   ```

**User Workflow:**
```bash
# Dry run — validate without executing
$ cob-shopify products create --title "New Widget" --dry-run
{"action":"create_product","domain":"products","input":{"title":"New Widget"},"confirmed":false}

# Interactive confirmation
$ cob-shopify products delete --id gid://shopify/Product/123
⚠ This will delete products. Continue? [y/N] y
# (executes)

# Skip confirmation
$ cob-shopify products delete --id gid://shopify/Product/123 --yes
# (executes immediately)

# Non-TTY (piped) — no prompt
$ echo '{"id":"gid://shopify/Product/123"}' | cob-shopify products delete --yes
```

**Risk Assessment:** Low. This adds safety, not risk. The only concern is false negatives — a mutation tool without `write_*` scope would bypass safety checks. But all Shopify mutation tools correctly declare write scopes.

## 4. Scope Guardrails

**In Scope:**
- Mutation detection from tool scopes
- `--dry-run` handler with Zod validation and JSON output
- Interactive confirmation prompt using Node.js readline
- `--yes` flag bypass
- Auto-skip prompt when not TTY

**Out of Scope:**
- Showing the GraphQL query in dry-run output (nice-to-have for later)
- Confirmation for read operations (reads never prompt)
- Custom confirmation messages per tool (generic message based on action + domain)

## 6. Acceptance Criteria

- [ ] `--dry-run` on a write tool validates input and returns JSON without calling Shopify
- [ ] `--dry-run` on a read tool shows validated input (no harm, just informational)
- [ ] Running a write tool in TTY without `--yes` shows confirmation prompt
- [ ] Answering "n" or empty to confirmation aborts with exit code 0 and message
- [ ] Answering "y" proceeds with execution
- [ ] `--yes` skips confirmation
- [ ] Non-TTY (piped) auto-skips confirmation (behaves as `--yes`)
- [ ] Read tools (no `write_*` scopes) never prompt for confirmation
- [ ] Dry-run output includes action, domain, input, confirmed:false, message
