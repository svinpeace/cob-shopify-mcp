## 1. Executive Context

**Product Intent:** The CLI redesign requires a `cob-shopify <domain> <action>` pattern. Every tool name (e.g., `list_products`) must be automatically converted to an action name (e.g., `list`) by stripping the domain prefix. Additionally, every tool's Zod input schema must be convertible to citty CLI argument definitions so flags like `--limit`, `--status` are auto-generated.

**System Position:** New utility module at `src/cli/converter/` used by `toolToCommand()` (CLI-003) and domain registration (CLI-004). Depends only on types from `src/core/engine/types.ts` and `zod`.

## 2. Feature Specification

**What Exists Today:**
- `ToolDefinition` interface at `src/core/engine/types.ts:8-18` with `name`, `domain`, `input: Record<string, ZodType>`
- `citty` v0.2.1 is already a dependency (`package.json:32`)
- citty uses `defineCommand({ args: { flagName: { type: "string"|"boolean"|"positional", description, required, default } } })`

**What Must Be Built:**
1. `deriveActionName(toolName: string, domain: string): string` — strips domain singular/plural from tool name, replaces `_` with `-`
2. `zodToCittyArgs(input: Record<string, ZodType>): Record<string, CittyArg>` — converts Zod schemas to citty arg definitions
3. `zodToJsonSchema(input: Record<string, ZodType>): Record<string, JsonSchemaField>` — serializes Zod schemas to JSON (for `--describe`, used in CLI-005)

**User Workflow:** No direct user interaction — these are internal utilities. Used by toolToCommand() to auto-generate CLI commands.

**Risk Assessment:** Low. Pure functions with no side effects. Wrong name derivation would produce bad CLI commands but is caught by tests.
