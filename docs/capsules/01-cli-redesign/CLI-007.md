# CLI-007: Collision Handling in ToolRegistry

As a user, I cannot accidentally create a custom tool with a reserved domain name that would break CLI routing.

## 1. Executive Context

**Product Intent:** Custom YAML tools are first-class citizens. When a custom tool has the same name as a built-in, the custom version wins (override pattern). This applies to both MCP and CLI since they share the same `ToolRegistry`. Additionally, certain domain names are reserved for CLI routing commands and must be rejected.

**System Position:** Modifies `src/core/registry/tool-registry.ts` (the `register()` method). Affects both MCP and CLI registration paths since both use `ToolRegistry`.

## 2. Feature Specification

**What Exists Today:**
- `ToolRegistry.register()` at `src/core/registry/tool-registry.ts:7-11` throws an error if a tool with the same name is already registered:
  ```typescript
  register(tool: ToolDefinition): void {
      if (this.tools.has(tool.name)) {
          throw new Error(`Tool "${tool.name}" is already registered`);
      }
      this.tools.set(tool.name, tool);
  }
  ```
- Custom YAML tools are registered AFTER built-in tools (see `src/cli/commands/tools/run.ts:38-49`)

**What Must Be Built:**
1. Modify `register()` to support override mode:
   - If tool name already exists AND new tool is tier 3 (custom), override with warning
   - If tool name already exists AND new tool is NOT tier 3, throw (built-in collision = bug)
2. Add `RESERVED_DOMAINS` constant: `["start", "connect", "config", "tools", "stores"]`
3. Add domain validation: reject tools with reserved domains via a `validateDomain()` check
4. Add a logger parameter or callback for warnings (currently ToolRegistry has no logger)

**User Workflow:**
```yaml
# User creates custom-tools/list_products.yaml
name: list_products
domain: products
description: "My custom product listing with extra fields"
graphql: |
  query { ... }
```
On startup: `⚠ Custom tool "list_products" overrides built-in tool` — custom version is used.

```yaml
# User creates custom-tools/my_config.yaml
name: my_config
domain: config    # RESERVED!
```
On startup: `✗ Domain "config" is reserved. Skipping custom tool "my_config".`

**Risk Assessment:** Low. The change is backward-compatible — built-in tools are registered first and custom tools come after, so the override direction is always custom→built-in.
