## 1. Executive Context

**Product Intent:** With 49+ built-in tools (growing to 200+), runtime registration at every CLI invocation wastes startup time. Pre-generating command definitions at build time (via tsup) eliminates this cost. Custom YAML tools still register at runtime using the same `toolToCommand()` function — same code path, different timing.

**System Position:** Adds a tsup plugin or build script that imports all tool definitions, runs `toolToCommand()`, and writes a generated file with pre-built command definitions. The CLI entry point imports the generated file for built-ins and only runs `toolToCommand()` at runtime for custom YAML tools.

## 2. Feature Specification

**What Exists Today:**
- `tsup.config.ts` at root — current build config with `src/index.ts` and `src/cli/index.ts` entry points
- `toolToCommand()` from CLI-003 runs at runtime for all tools
- Build takes ~2-3 seconds

**What Must Be Built:**
1. Build script or tsup plugin that:
   - Imports `getAllTools()` from `src/server/get-all-tools.ts`
   - Runs `toolToCommand()` for each built-in tool
   - Serializes the citty command definitions to a generated file
   - Outputs `src/cli/generated/domain-commands.ts` (or similar)
2. Modify `src/cli/index.ts` to import generated commands for built-ins
3. Runtime path for custom YAML tools remains unchanged
4. Generated file is `.gitignore`d (not committed) and rebuilt on every `pnpm build`

**User Workflow:** No user-facing change. CLI starts faster, especially as tool count grows past 100.

**Risk Assessment:** Medium. Build-time code generation adds complexity to the build pipeline. Generated file must stay in sync with tool definitions. If build step fails, CLI has no built-in tools.

## 4. Scope Guardrails

**In Scope:**
- Build script that generates command definitions
- Generated file output
- CLI entry point modification to use generated file
- Fallback: if generated file missing, fall back to runtime registration
- `.gitignore` entry for generated file

**Out of Scope:**
- Changing `toolToCommand()` itself (already built in CLI-003)
- Changing custom YAML tool loading (already runtime)
- Hot module reload or watch mode for generated commands
- Performance benchmarking (premature — benefit is future-proofing for 200+ tools)
